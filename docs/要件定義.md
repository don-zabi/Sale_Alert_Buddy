# Sale Alert Buddy — 要件定義書

**バージョン:** 1.1
**作成日:** 2026-03-27
**更新日:** 2026-03-27（通知条件修正・データモデル見直し・SPA非対応明記・BG制限追加）
**対象:** iOS アプリ（サーバーなし・端末完結）

---

## 1. プロダクト概要

ユーザーが登録した商品ページURLを定期的にチェックし、**登録時の価格より安くなったらローカル通知を送る** iOSアプリ。サーバー不要・個人情報非収集・端末内完結。

---

## 2. 目標と成功条件

### 2.1 目標
- 値下げを見逃しにくくする
- サーバー運用なしで低コスト・低リスクで提供する
- 取得失敗（ブロック・構造変更）を前提に、UXが破綻しない設計にする

### 2.2 v1 成功条件
- URL登録 → 価格抽出 → 保存 → 値下げ判定 → ローカル通知が一連で動作する
- アプリ起動時に登録URLを（最大50件）チェックできる
- 取得失敗を「状態」として表示し、原因追跡できるログを保持する
- 無料（1件）・有料（50件）の上限を実装する

---

## 3. 非目標（v1ではやらない）

- バックグラウンド実行の「毎日必ず動く」保証（iOS制約のため不可能）
- サーバー経由のプッシュ通知（APNs）
- ログインが必要なサイトへの対応（Cookie認証など）
- **URL変化時の再紐づけ（Re-linking）** → **v1.1以降に延期**
- 規約違反を助長するBot回避行為（大量プロキシ・CAPTCHA回避など）
- **JavaScriptレンダリングが必要なSPAサイト**（メルカリ等 React/Next.js 製）への対応 → URLSessionは静的HTMLのみ取得可能なため対象外

---

## 4. ユーザーとプラン

### 4.1 想定ユーザー
- 個人ユーザー（端末1台）
- 1日〜1週間に1回以上はアプリを開く運用が前提

### 4.2 プラン構成

| プラン | 登録上限 | 価格 |
|--------|---------|------|
| 無料 | 5件 | 無料 |
| 有料（サブスク） | 50件 | 月額・年額（StoreKit 2） |

**ダウングレード時:** 6件目以降のアイテムを `PAUSED`（`pauseReason = planLimit`）状態にする（データは削除しない）。

**再アップグレード時:** `pauseReason = planLimit` のアイテムを `createdAt` の古い順に自動的に `OK` へ遷移する。

---

## 5. 重要な制約

| 制約 | 内容 |
|------|------|
| バックグラウンド実行 | BGTaskSchedulerはベストエフォート。実行タイミング・頻度はOS裁量 |
| Bot対策 | 対象サイトにレート制限・アクセス制限が存在しうる |
| HTML構造 | 価格抽出は壊れる前提で複数戦略でフェイルオーバーする |
| ベースライン価格 | **登録時価格に固定**（通知後も更新しない） |
| 通知重複抑制 | 同一アイテム・同一価格での再通知は行わない |

---

## 6. 機能要件

### 6.1 URL登録

**入力項目:**
- URL（必須）
- メモ（任意）
- タグ（任意）

**登録時処理（順序）:**
1. URL正規化（`utm_*` などのトラッキングパラメータ除去・末尾スラッシュ除去）
2. HTMLフェッチ（初回）
3. 価格抽出（複数戦略でフェイルオーバー）
4. 商品名抽出（`title` / `og:title`）
5. 正規URL抽出（`canonical` / `og:url`）
6. 追跡キー抽出（SKU・GTIN・型番に相当する文字列）
7. 代表画像URL抽出（`og:image`）
8. **ベースライン価格として保存**（以後変更しない）

**登録失敗時:**
- 失敗理由（ネットワーク / HTTP エラー / 抽出不能）を保存
- 状態を `TEMP_FAILED` にしてUIに明示

---

### 6.2 一斉チェック

**トリガー:**
- アプリ起動時・フォアグラウンド復帰時（主軸）
- ユーザーによる手動更新ボタン
- BGTaskScheduler によるバックグラウンド更新（ベストエフォート）

**実行仕様:**
- 対象: 状態が `OK` または `TEMP_FAILED` のアイテム（`PAUSED` は除外）
- 同時実行数: 最大3〜5件並列
- タイムアウト: 15秒/URL
- ドメイン別スロットリングを適用（後述）
- 結果を都度Core Dataに保存し、UIに反映

---

### 6.3 値下げ判定と通知

**判定条件（AND条件）:**
1. `latestPrice < baselinePrice`（ベースラインより安い）
2. `lastNotifiedPrice IS NULL` または `latestPrice < lastNotifiedPrice`（さらに値下がった場合のみ再通知。価格が戻った場合は通知しない）
3. 通貨がベースラインと一致する
4. `(baselinePrice - latestPrice) / baselinePrice >= notificationThreshold`（値下がり率が閾値以上。デフォルト: 1%、ユーザー設定可）

**無効ケース（通知しない）:**
- 価格が0または異常値（¥10,000,000超など）
- 通貨が取得できない

**ローカル通知内容:**

| 項目 | 内容 |
|------|------|
| タイトル | `{商品名} が値下がりしました` / `Price Drop` |
| 本文 | `{ベースライン価格} → {最新価格}（{ドメイン}）` |
| タップ時 | アプリの詳細画面へ遷移 |

---

### 6.4 取得失敗の扱い

**失敗種別の分類:**

| 種別 | 例 |
|------|-----|
| `network` | オフライン・DNS失敗・タイムアウト |
| `http4xx` | 403 / 404 / 410 / 429 など |
| `http5xx` | 500 / 503 など |
| `extractionFailed` | HTML取得成功したが価格を見つけられない |

**連続失敗時の状態遷移:**
- 失敗ごとに `failCountConsecutive` をインクリメント
- 3回連続失敗 → `PAUSED`（自動停止）
- 429 / 403 受信 → 即 `PAUSED` + ドメインバックオフ強化

---

### 6.5 URL変化への対応（Re-linking）

> **v1では実装しない。v1.1以降に延期。**

v1での対応:
- 404 / 410 または抽出失敗連続 → `PAUSED` に遷移
- UI上で「このURLへのアクセスが失敗しています。URLを再入力してください」と表示
- ユーザーが手動で新しいURLを入力して再登録

## 7. 状態マシン

### 状態一覧（v1）

| 状態 | 説明 |
|------|------|
| `OK` | 正常監視中 |
| `TEMP_FAILED` | 一時失敗（次回再試行） |
| `PAUSED` | 停止中（自動チェックしない） |

> `NEEDS_RELINK` / `RELINK_CANDIDATE_READY` は v1.1以降で追加予定。

### 主要な遷移ルール

```
取得成功          → OK（failCountConsecutive リセット）
取得失敗          → TEMP_FAILED（failCountConsecutive +1）
3回連続失敗       → PAUSED
429 / 403 受信    → PAUSED（即時）+ ドメインバックオフ
ユーザー手動停止  → PAUSED
ユーザー再開      → OK（再チェック実行）
```

## 9. 価格抽出アルゴリズム

### 9.1 フェッチ方法
- `URLSession` でHTMLをGET取得（タイムアウト15秒）
- User-Agent: Safariモバイルに準拠した識別文字列（偽装ではなく明示）
- リダイレクト追跡（最大回数制限あり）

### 9.2 抽出戦略（優先順・フェイルオーバー）

| 優先度 | 戦略 | 対象 |
|--------|------|------|
| 1 | `SchemaOrgExtractor` | JSON-LD `@type: Product/Offer` の `price`, `priceCurrency` |
| 2 | `MetaTagExtractor` | `og:price:amount`, `product:price:amount` |
| 3 | `DataAttributeExtractor` | `data-price`, `data-product-price`, `data-amount` |
| 4 | `HTMLPatternExtractor` | 通貨記号+数字のRegex + `.price`/`#price` CSS class |
| — | 失敗 | `extractionFailed` として記録 |

### 9.3 妥当性チェック
- 価格 = 0 → 無効
- 異常値上限（通貨別）:
  - JPY: ¥10,000,000 超 → 要確認フラグ
  - USD / EUR: $100,000 超 → 要確認フラグ
  - 上記以外の通貨: 取得価格をそのまま使用（上限チェックなし）
- 通貨不明 → 通知しない
- 複数候補がある場合 → スコアリングで最有力候補を採用（最安値を安易に採用しない）

---

## 10. ドメイン別スロットリング

### 目的
同一ドメインへのアクセス集中によるブロック悪化を防ぐ。

### 仕様

| 設定値 | デフォルト |
|--------|-----------|
| `minIntervalSeconds` | 2.0秒 |
| `backoffSeconds` | 0（エラーで増加） |
| 429受信時 | `backoffSeconds × 2`（上限: 86400秒=24時間） |
| 403が同一ドメインで3回連続 | ドメイン配下のアイテム全て `PAUSED` |
| 成功時 | `backoffSeconds` リセット |

---

## 11. バックグラウンド実行

- `BGAppRefreshTask`（ID: `com.salealertbuddy.refresh`）を使用
- 目標間隔: 2時間（iOSの裁量で決定。保証なし）
- バックグラウンド予算: 約25秒以内に完了
- **バックグラウンド実行時の制限（フォアグラウンドとは別設定）:**
  - 同時実行数: 最大2〜3件（フォアグラウンドの5件より少なく制限）
  - タイムアウト: 8秒/URL（フォアグラウンドの15秒より短く設定）
  - 理由: BGTaskSchedulerの~30秒killタイムリミット内に確実に収める
- 優先度: `lastCheckedAt` が古いアイテム優先
- 完了後に次回タスクをスケジュール

## 14. 通知仕様

### 通知種類

| 種類 | 条件 |
|------|------|
| 値下げ通知 | `latestPrice < baselinePrice` かつ重複でない |
| 停止通知（任意） | 連続失敗でPAUSEDになったとき |

### 重複抑制
- `lastNotifiedPrice IS NULL` または `latestPrice < lastNotifiedPrice` の場合のみ通知
- 一度通知後に価格が戻った（値上がった）場合は再通知しない
- さらに価格が下がった（`latestPrice < lastNotifiedPrice`）場合のみ再通知

### 夜間通知制御
v1.1で実装（UserDefaultsで時間帯設定）

## 15. セキュリティ・プライバシー

- 収集データは端末内に限定
- ログイン情報（ID/PW/Cookie）を保存しない
- 通信はHTTPSのみ（ATS準拠）
- 解析SDK・広告SDKは使用しない（v1）
- URLは個人の嗜好情報になりうるため、必要に応じてDB暗号化を評価

## 17. テスト要件

### ユニットテスト
- URL正規化（Amazon, 楽天, Yahoo!ショッピングのURL形式）
- 価格抽出（各サイトのHTMLスナップショットを使用）

### 統合テスト

### 手動・デバイステスト
- BGTaskScheduler: Xcode LLDBコマンドで動作確認
- StoreKit: Xcodeサンドボックスで購入フロー確認
- 通知: フォアグラウンド・バックグラウンド・強制終了後の受信確認

## 22. 技術スタック

| 項目 | 採用技術 | 理由 |
|------|---------|------|
| UI | SwiftUI | 宣言的・ネイティブ |
| アーキテクチャ | MVVM + `@Observable`（iOS 17） | TCAより軽量・このスコープに適切 |
| 永続化 | Core Data | 仕様指定・実績ある選択肢 |
| ネットワーク | URLSession + async/await | ネイティブ・依存ゼロ。**静的HTMLのみ対応（JSレンダリングなし）** |
| HTMLパース | SwiftSoup（SPM） | Pure Swift・CSS セレクタAPI |
